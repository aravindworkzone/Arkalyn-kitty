# Project: Expense Tracker

## What this project does
A MERN-stack group expense-sharing app built on a pooled wallet model. Group members contribute money to a shared balance, and expenses are debited from that pool — eliminating circular debt tracking. Used by small groups (friends, roommates, teams) to manage shared spending transparently, with per-group roles, categories, and an immutable audit trail.

## Tech Stack
- **Frontend:** React 19 + Vite + Tailwind CSS + TypeScript
- **Backend:** Node.js + Express 5 + TypeScript
- **DB:** MongoDB (Atlas) + Mongoose
- **Auth:** JWT in HTTP-only cookies (single token, no refresh rotation)
- **State:** Redux Toolkit + RTK Query (caching, fetching, tag invalidation)

## Architecture / Request Flow

```
client → Express Router → Middleware (verifyToken → loadGroup → authorizeRole)
       → Controller (parse req, format res)
       → Service (all business logic + constraint validation)
       → Model (Mongoose schema + DB queries)
       → MongoDB (sessions for multi-doc writes)
```

Service layer is mandatory — controllers must not query models directly.

**Frontend flow:**
```
React page → RTK Query hook → base query (sends cookies) → Backend API
          ← cache invalidation via tag types (Auth, Group, Expense, Category, User)
          ← auto-redirect to /login on 401
```

## Folder Structure

```
Frontend/src/
  redux/api/       # RTK Query endpoint files per feature
  redux/store.ts   # Single store — RTK Query is the only reducer
  page/            # Route-level page components
  components/      # Reusable UI components (components/ui = design-system primitives)
  hooks/           # Reusable hooks (useApiError = RTK error normalizer, useFieldError, usePlan, socket)
  interface/       # TypeScript interfaces
  utils/           # Helpers
  App.tsx          # Route definitions + ProtectedRouter + ErrorBoundary + TopProgressBar

backend/
  config/          # env.ts (validateEnv, typed env), constants.ts (JWT, cookie, pagination, rate-limit values)
  controllers/     # Request parsing and response formatting only
  db/              # MongoDB connection (exponential backoff, max 5 retries)
  helpers/         # Money.ts (cents conversion), AppError.ts (extends Error)
  middlewares/     # auth.middleware.ts (verifyToken, loadGroup, authorizeRole), error.middleware.ts, validate.ts
  models/          # Mongoose schemas
  routes/          # Express route definitions per feature
  services/        # Business logic, validations, balance checks
  sockets/         # Socket.io setup (group rooms, typed events) — scaffold, not yet emitting
  types/           # TypeScript type extensions (e.g., req.group, req.user) + DTOs
  utils/           # response.ts (sendSuccess/sendPaginated/sendError), asyncHandler, logger (pino)
  validators/      # Zod schemas per feature (auth, group, expense, category)
```

## Key Conventions
- Always use `async/await`, never `.then()`
- All responses follow `{ success, message, data }` shape — use `AppError` for thrown errors
- Money is stored as **integer cents** in DB — use `toDBAmount()` before saving, `fromDBAmount()` before responding; never arithmetic on raw decimal values
- Expenses are **immutable** — no PATCH endpoint; delete and recreate to correct
- Every balance mutation must write to both `GroupTransaction` (audit) and `GroupEvent` (admin log)
- Soft deletes only — `GroupMember` and `Category` use `isDeleted: true`; always filter these out in queries
- Group `displayId` (e.g., `Grp-25-001`) is generated by the `Counter` sequence model — never set it manually
- Frontend API errors: normalize RTK Query errors with `getApiErrorMessage` / `useApiError` (`hooks/useApiError.ts`) — don't reach into `error.data.message` per call site
- Admin analytics / dashboard aggregations must scale: bucket + count in MongoDB (`$group`), never load all users/rows into Node memory

## Environment Variables

See `backend/.env.example` and `frontend/.env.example` for the full list with comments.

**Backend `.env`:**
- Required: `PORT`, `MONGO_URI`, `ACCESS_TOKEN_SECRET`, `REFRESH_TOKEN_SECRET`, `FRONTEND_URL`
- Optional: `NODE_ENV`, `ACCESS_TOKEN_EXPIRES_IN`, `REFRESH_TOKEN_EXPIRES_IN`

**Frontend `.env`:**
- `VITE_API_URL` (e.g. `http://localhost:5000/api`)

Never hardcode. Always `process.env` (backend) or `import.meta.env` (frontend).
Required vars are validated on startup by `validateEnv()` in `config/env.ts`.

## What NOT to do
- Don't bypass the Service layer — controllers must not call Mongoose models directly
- Don't change the structure without also updating this file — folder layout is documented above
- Don't use `var` — only `const`/`let`
- Don't do multi-document writes without a Mongoose session (transaction)
- Don't add a PATCH endpoint for expenses — the design is intentionally immutable
- Don't query `GroupMember` or `Category` without filtering `isDeleted: false`
- Don't store money as floats — always convert to cents before saving

## Testing
- No automated test suite. Backend type-checks via `npm run build` (tsc); frontend via `npx tsc --noEmit -p tsconfig.json` + `npm run lint` (ESLint).
- Manual / API testing via Postman or browser devtools.
- ⚠️ The server runs compiled `dist/` (`npm start` → `node dist/main.js`). After backend edits you must `npm run build` **and restart** — or run `npm run dev` (ts-node, auto-reloads). Editing a `.ts` file alone does NOT change a running `dist` server.

## Notable behaviors (non-obvious)
- **Admin hard-delete transfers ownership** (`hardDeleteUserService`): in one transaction, sole-member groups are cascade-deleted; owned groups with survivors hand `SUPER_ADMIN` to the earliest-joined `ADMIN` (else earliest `MEMBER`); the membership row is removed but contribution credits stay in `GroupTransaction` (render as "Deleted user"); expenses are kept; the promoted member is notified after commit.
- **Reports are plan-gated**: `this_month` / `last_month` are free; all-time and custom ranges require Pro/Premium (`requireAdvancedReportRange` → 402).
- **Member breakdown `spent` attribution** (`memberBreakdownService`): "Who spent" attributes split expenses by each member's share AND unsplit expenses in full to their payer (split tracking is opt-in, so most rows are unsplit). This keeps the `spent` total equal to the `paid` total and matches the expense-list `spender` filter (`getAllExpensesService`: `splitBetween.userId` OR `paidBy` on unsplit rows) so the chart and its drill-down reconcile — keep the two in sync.
- **Admin analytics** (`getAnalyticsService`) buckets + counts in MongoDB, then reuses `getEffectivePlan` / `monthlyEquivalent` for MRR — keep both in sync if pricing/expiry logic changes.
- Admin routes are APP_OWNER-only (`verifyToken` + `requireAppOwner`); a normal USER gets 403.
