# Project: Expense Tracker

## What this project does
A MERN-stack group expense-sharing app built on a pooled wallet model. Group members contribute money to a shared balance, and expenses are debited from that pool ŌĆö eliminating circular debt tracking. Used by small groups (friends, roommates, teams) to manage shared spending transparently, with per-group roles, categories, and an immutable audit trail.

## Tech Stack
- **Frontend:** React 19 + Vite + Tailwind CSS + TypeScript
- **Backend:** Node.js + Express 5 + TypeScript
- **DB:** MongoDB (Atlas) + Mongoose
- **Auth:** JWT in HTTP-only cookies (single token, no refresh rotation)
- **State:** Redux Toolkit + RTK Query (caching, fetching, tag invalidation)

## Architecture / Request Flow

```
client ŌåÆ Express Router ŌåÆ Middleware (verifyToken ŌåÆ loadGroup ŌåÆ authorizeRole)
       ŌåÆ Controller (parse req, format res)
       ŌåÆ Service (all business logic + constraint validation)
       ŌåÆ Model (Mongoose schema + DB queries)
       ŌåÆ MongoDB (sessions for multi-doc writes)
```

Service layer is mandatory ŌĆö controllers must not query models directly.

**Frontend flow:**
```
React page ŌåÆ RTK Query hook ŌåÆ base query (sends cookies) ŌåÆ Backend API
          ŌåÉ cache invalidation via tag types (Auth, Group, Expense, Category, User)
          ŌåÉ auto-redirect to /login on 401
```

## Key Conventions
- Always use `async/await`, never `.then()`
- All responses follow `{ success, message, data }` shape ŌĆö use `AppError` for thrown errors
- Money is stored as **integer cents** in DB ŌĆö use `toDBAmount()` before saving, `fromDBAmount()` before responding; never arithmetic on raw decimal values
- Expenses are **editable in place** (`PATCH /expense/update/:id` ŌåÆ `updateExpenseService`). Editing is allowed for admins **or the expense's own payer** (gated inside the service, not via `authorizeRole`). The amount delta is applied to the pool atomically (`debitGroupBalance` when Ōåæ with the `$gte` guard, `refundGroupBalance` when Ōåō) and every edit writes an `EXPENSE_EDITED` `GroupEvent` with a beforeŌåÆafter diff (the **edit log**, shown in the Activity tab). Deletes are still soft-delete + refund.
- Every balance mutation must write to both `GroupTransaction` (audit) and `GroupEvent` (admin log)
- Soft deletes only ŌĆö `GroupMember` and `Category` use `isDeleted: true`; always filter these out in queries
- Group `displayId` (e.g., `Grp-25-001`) is generated by the `Counter` sequence model ŌĆö never set it manually
- Frontend API errors: normalize RTK Query errors with `getApiErrorMessage` / `useApiError` (`hooks/useApiError.ts`) ŌĆö don't reach into `error.data.message` per call site
- Admin analytics / dashboard aggregations must scale: bucket + count in MongoDB (`$group`), never load all users/rows into Node memory

## What NOT to do
- Don't bypass the Service layer ŌĆö controllers must not call Mongoose models directly
- Don't change the structure without also updating this file ŌĆö folder layout is documented above
- Don't use `var` ŌĆö only `const`/`let`
- Don't do multi-document writes without a Mongoose session (transaction)
- Don't query `GroupMember` or `Category` without filtering `isDeleted: false`
- Don't store money as floats ŌĆö always convert to cents before saving

## Notable behaviors (non-obvious)
- **Group purpose + seeded categories** (`createGroupService`): a group has a `purpose`
  (`FAMILY`/`FRIENDS`/`ROOMMATES`/`TEAM`/`OTHER`, default `OTHER`). On creation the purpose's
  default categories (`config/purposeCategories.ts`) are `insertMany`-seeded **inside the creation
  transaction**, bypassing the per-group category plan limit (same spirit as clone). `OTHER` seeds none.
- **Special / "collective" categories** (`Category.isSpecial`): the FAMILY purpose seeds a special
  `Family` category (for shared obligations like an EMI); admins can also toggle `isSpecial` on any
  category via `PATCH /category/update/:id` (the same endpoint that edits colour ŌĆö body accepts
  `color?` and/or `isSpecial?`). Special categories are **excluded from the per-member breakdown**
  (`memberBreakdownService` adds `category: { $nin: specialIds }` to the match for both members and
  the total) and returned as a separate `specialCategories` bucket. **Keep the drill-down in sync**:
  `getAllExpensesService` excludes special categories when filtering by `spender`/`paidBy` *without* a
  `categoryId` (member-chart drill-down), but shows them when a `categoryId` is explicitly requested
  (collective-bucket / category-report / category-page deep-links). Category report, spend-trend, and
  the group total still include special categories normally.
- **Expense edit** (`updateExpenseService`): in-place update inside one Mongoose session. The expense keeps its `_id`/`createdAt`. Authorization is admin-or-payer (checked in the service via the requester's `GroupMember.role` OR `expense.paidBy`). Only the **amount delta** touches the pool ŌĆö a `GroupTransaction` (`DEBIT`/`REFUND`) is written **only when the amount changed**, while an `EXPENSE_EDITED` `GroupEvent` is **always** written with `metadata.note` (summary) + `metadata.changes` (per-field `{from,to}` diff, with category/user ids resolved to names). The frontend reuses `CreateExpense.tsx` for both create and edit (route `/groups/:groupId/expenses/:expenseId/edit`); `getExpenseByIdService` (`GET /expense/one/:groupId/:id`) prefills the form so a hard refresh still works. Socket event: `EXPENSE_UPDATED`.
- **Modal centering convention** (all overlay dialogs): the overlay is `fixed inset-0 flex justify-center overflow-y-auto p-4` and the dialog is `relative my-auto w-full max-w-ŌĆ” max-h-[88dvh]`. `justify-center` centers horizontally; `my-auto` centers vertically *and* lets tall content scroll instead of clipping. Do NOT use `items-center justify-center` (top-clips on short/mobile viewports and behind the iOS address bar) and do NOT omit `justify-center` (the capped-width dialog then pins to the left on wide screens). Use `dvh` not `vh` for mobile max-heights. Every modal also locks background scroll (`document.body.style.overflow = 'hidden'` on open). The shared `components/ui/BottomSheet` already follows this; new modals must too.
  - ŌÜĀ’ĖÅ **`position: fixed` needs a transform-free ancestor chain.** Any ancestor with a non-`none` `transform`/`filter`/`perspective`/`will-change`/`contain` becomes the containing block, so the modal centers against *that element* (page-sized), not the viewport ŌĆö looking like `absolute`. The app's page-transition wrapper `.route-fade` is therefore kept **transform-free** (opacity-only `routeFade` keyframes in `index.css`). Don't add a transform to it or to any layout wrapper that contains modals.
- **Global horizontal-overflow guard**: `index.css` sets `body { overflow-x: clip }` ŌĆö **`body` only, never `html`**. While `<html>`'s overflow stays `visible`, the body's overflow propagates to the viewport, which (a) blocks sideways scroll and (b) keeps the modals' `body { overflow: hidden }` working as a real scroll lock. Putting any overflow on `<html>` makes `<html>` the propagating root and silently disables that lock. `clip` (not `hidden`) avoids a scroll container so the sticky header works. It's a safety net ŌĆö still use `min-w-0` + `truncate`/`break-all` on flex rows with user text rather than relying on it.
- **Admin hard-delete transfers ownership** (`hardDeleteUserService`): in one transaction, sole-member groups are cascade-deleted; owned groups with survivors hand `SUPER_ADMIN` to the earliest-joined `ADMIN` (else earliest `MEMBER`); the membership row is removed but contribution credits stay in `GroupTransaction` (render as "Deleted user"); expenses are kept; the promoted member is notified after commit.
- **Reports are plan-gated**: `this_month` / `last_month` are free; all-time and custom ranges require Pro/Premium (`requireAdvancedReportRange` ŌåÆ 402).
- **No subscription downgrade while active**: `createSubscriptionOrderService` rejects buying a strictly lower tier when the buyer's effective plan is `active`/`grace` (via `PLAN_RANK`, mirroring the promo guard) ŌåÆ 400; same-tier renewal + upgrades allowed. The pricing page also disables the lower-tier card.
- **Payment outcome is event-driven**: each checkout attempt is one `SubscriptionPayment` (`createdŌåÆpaid` on verify; `createdŌåÆfailed` via `POST /subscription/payment-failed` on Razorpay `payment.failed` or a dismissed modal ŌĆö never overwrites `paid`). `grantFromPayment`'s lock filter is `status Ōłł {created, failed}` so a capture webhook still grants an attempt the browser optimistically failed. `GET /subscription/transactions` powers the profile "Subscription transactions" list (Pending/Success/Failed).
- **Member breakdown `spent` attribution** (`memberBreakdownService`): "Who spent" attributes split expenses by each member's share AND unsplit expenses in full to their payer (split tracking is opt-in, so most rows are unsplit). This keeps the `spent` total equal to the `paid` total and matches the expense-list `spender` filter (`getAllExpensesService`: `splitBetween.userId` OR `paidBy` on unsplit rows) so the chart and its drill-down reconcile ŌĆö keep the two in sync.
- **Admin analytics** (`getAnalyticsService`) buckets + counts in MongoDB, then reuses `getEffectivePlan` / `monthlyEquivalent` for MRR ŌĆö keep both in sync if pricing/expiry logic changes.
- Admin routes are APP_OWNER-only (`verifyToken` + `requireAppOwner`); a normal USER gets 403.
- **Per-user API keys + MCP** (`/api/mcp/*`, `arkalyn-mcp/`): a user generates one personal key
  (`POST /api/user/generate-api-key`, JWT-cookie auth) of the form `ak_live_` + 32 url-safe random
  chars. Only the **bcrypt hash** is stored (`User.apiKey`, `select:false`); the plaintext is returned
  **once** and never recoverable. `User.apiKeyPrefix` keeps the first 16 chars (`ak_live_` + 8 random)
  in plaintext ŌĆö used both for the masked profile display and as an **indexed lookup filter**, since a
  bcrypt hash can't be queried directly. `apiKeyAuth.middleware.ts` reads `x-api-key`, narrows by
  prefix, then `bcrypt.compare`s each candidate; sets `req.user = { _id, role }` like `verifyToken`.
  `DELETE /api/user/revoke-api-key` clears all three fields (idempotent). The `/api/mcp/*` surface is
  **GET-only and read-only by design** (`mcp.service.ts` performs no writes) and **derives groups from
  the caller's own memberships** ŌĆö never a client-supplied `groupId`. Four endpoints: `balance`,
  `expenses?limit=` (clamped 1ŌĆō`MAX_LIMIT`, default 10), `members`, `subscription`. `GET /user/me` also
  returns a masked `apiKey: { prefix, createdAt } | null` so the Profile ŌåÆ Developer section can show
  Generate vs. masked-prefix+Revoke. The standalone `arkalyn-mcp` server passes the user's key as
  `x-api-key` on every (GET-only) `callAPI`; its `POST /messages` route is the MCP SSE protocol channel,
  not a data write.
